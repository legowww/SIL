>>TCP와 UDP의 차이점에 대해서 설명해보세요.
	티시피와 유디피는 모두 패킷 교환 방식이라는 공통점이 존재하지만 
	연결지향형과 신뢰성에서 차이가 있습니다.
	
	티시피는 가상 회선을 맺은 후 페이로드를 전송할 수 있는 연결지항형 방식입니다.
	반면 유디피는 연결없이 즉시 페이로드를 전송할 수 있습니다. 
	그러므로 유디피가 더 빠르게 데이터를 전송할 수 잇어
	DNS, DHCP 프로토콜의 경우 속도를 위해 유디피가 사용됩니다.
	
	티시피는 신뢰성 통신을 보장합니다. 
	패킷에 오류가 있거나 유실됐을 경우 패킷 재전송을 수행할 수 있습니다.
	(체크섬을 통해 ACK 0 Flag)
	(ACK 오지않으면 타임아웃 재전송)
	또한 흐름제어와 혼잡제어를 사용하여 패킷이 큐 공간을 넘치지 않게 전송량을 조절할 수 있습니다.
	
	하지만 유디피는 신뢰성을 보장하지 않습니다. 그 이유는 유디피 프로토콜은
	네트워크 계층의 패킷인 IP데이터그램에 포트번호와 체크섬정도만 추가되 때문입니다.
	티씨피 프로콜의 경우 20바이트 크기의 헤더가 추가되는 반면 
	8바이트 크기의 헤더가 추가됩니다.


>>TCP 3, 4 way handshake에 대해서 설명해보세요.
	3way는 두 호스트가 연결을 수립할때, 4way는 연결종료 시 사용됩니다.

	tcp 소켓이 생성되면 3way가 동작합니다.
	클라이언트는 SYN세그먼트를 전송합니다. 이때 난수값을 가진 시퀸스넘버를 전송합니다.
	서버는 SYNACK세그먼트를 전송합니다. 서버도 난수값을 전송하며, ack넘버로 클라이언트의 시퀸스넘버+1값을 전송합니다.
	클라이언트는 ACK세그먼트를 전송합니다. 이때부터 연결이 수립되므로 페이로드를 함꼐전송할 수 있고, ack넘버로 서버의 시퀸스넘버+1값을. 포함하여 전송합니다.

	난수는 왜 설정하나요? -> 임의의 정수값은 0으로 변환되며, 3way를 통해 버퍼의 1바이트부터 전송할 수 있도록 설정하는 역할입니다.

	4way는 연결을 해제하는 단계입니다. 
	FIN 연결종료요청
	ACK 서버는 모든 데이터를 보내기위해 CLOSE_WAIT상태
	FIN 서버가 모든 데이터를 보냄
	ACK 확인의 의미로 ACK전송하고 TIME_WAIT상태를 가집니다.
	이후 소켓이 닫힙니다.
	단계로 구성됩니다.


>> HTTP와 HTTPS의 차이점에 대해서 설명해보세요.
	HTTP프로토콜은 TCP기반으로 만들어진 프로토콜입니다. TCP는 보안에 있어 특별한 기능이 없기떄문에
	전송되는 패킷을 도청할 경우 페이로드가 공격자에게 노출될 수 있습니다. 

	HTTPS는 HTTP프로토콜에 SSL소켓을 적용하여
	암호화된 http패킷을 전송하는 것입니다. 애플리케이션 계층에 존재하는 SSL 소켓은 평문으로 작성된 http패킷을 암호화한후 tcp 소켓으로 밀어내는 역할을 수행합니다.

>> GET과 POST의 차이점에 대해서 설명해보세요.
	GET은 리소스의 정보의 표현을 요청할때 사용되는 메서드이며, 
	POST는 리소스를 생성하거나, 리소스에 어떠한 프로세스를 동작시킬때 사용하는 메서드입니다.

	GET은 데이터를 URL의 쿼리스트링으로 전송합니다. HTTP 메세지 바디를 사용할 수 있지만, 일반적으로는 사용하지 않습니다.OPTIONS 메서드와 함께 캐싱에 주로 사용되는 메서드이며, 멱등성을 가지기때문에 1번요청하나 10번요청하나 동일한 결과가 똑같습니다.

	POST는 페이로드를 HTTP 메세지 바디에 담아 전송합니다. 그렇기 때문에 GET에 비해 보안적으로는 안전할 수 있습니다. 멱등성을 가지지 않습니다.

>> HTTP 메서드와 이것이 하는 역할에 대해서 설명해보세요.
	
	GET: 리소스의 표현을 요청할때 사용합니다. 
	POST: 리소스를 생성하거나 어떠한 프로세스를 수행하기 위해 사용합니다.
	PUT: 기존 리소스를 완전히 교체할때 사용합니다. 리소스 생성목적으로 사용하기도합니다. 
	PATCH: 리소스의 일부를 교체할때 사용합니다.
		- 제이슨 커맨드 패치
		- 제이슨 머지 패치
	DELETE: 리소스를 제거합니다.

	>> 패치와 풋 차이점
	리소스가 여러개의 필드로 구성됐다면, 풋의 경우 리소스 전체를 교체하고
	패치의 경우 일부 필드만 교체할 수 있습니다. 풋은 멱등성을 가집니다.

	유니폼인터페이스
	레이어드시스템이 존재합니다.

	캐시는 응답값이 캐시가 가능함을 의미합니다.
	클라이언트서버디커플링은 클라이언트는 API만 알면 되고, 서버는 응답만을 수행함을 의미합니다.
	유니폼인터페이스의 경우 서버가 제공하는 API는 리소스의 종류만 다르고 전체적으로 비슷한 형식의 API를 제공하는것을의미합니다.

	실제로 구현하는 경우 URI를 통해 리소스를 나타내고, HTTP메서드를 통해 리소스에 대한 행위를 표현합니다.

	웹브라우저는 access-control-allow 헤더를 읽어 응답값에 출처가 존재한다면 자원공유 허용
